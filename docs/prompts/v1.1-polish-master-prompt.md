# Master Prompt — V1.1: Polish (QA + UI/UX)

> **How to use this file.** Paste this entire document as the first message in a new Claude Code session with the Opus model, pointed at the `cost-seg` repo on the `main` branch. Work through the sub-phases in order; do not sprint past gates without review.

---

## 1. Mission

V1 shipped in nine phases and stands at commit `b3ccc69` on `main`. All
functional success criteria from the V1 master prompt §2 are satisfied:
estimator, Tier 1 end-to-end (Stripe → intake → AI pipeline → PDF →
delivery), Tier 2 manual admin handoff, scope-disclosed deliverables, admin
inspector, signed-URL delivery, app-layer authorization.

V1 is **functionally correct, visually minimal, and not yet polished**. The
UI is plain Tailwind on raw HTML. Forms, empty states, and error paths are
serviceable but not delightful. Accessibility has not been audited.
Loading/skeleton states are absent. Dark mode is mechanically supported but
visually uneven.

V1.1 is a **single-theme polish phase**. Two intertwined goals:

1. **QA pass** — a systematic review of every code path we've shipped,
   surfacing real defects, edge cases, and deferred cleanup. Output: a
   bug/risk register plus a set of fixes landed in the codebase.
2. **UI/UX overhaul** — stand up a disciplined design system, add the
   shadcn/ui primitive library, redesign every customer-facing surface for
   clarity and trust, and bring motion + microinteractions in. The app
   should feel like a product a CPA-referring tax advisor would happily
   hand to a client, not a demo.

Both goals land in parallel across the sub-phases in §4. QA findings that
block UI/UX work (e.g. error boundaries, Suspense fallbacks) ship first;
pure code quality fixes land alongside the surfaces they affect.

## 3. Scope of V1.1

**In scope**

- Systematic QA sweep: code review, security review, data-integrity
  review, accessibility audit, performance audit, copy audit. Output a
  written register and land targeted fixes.
- Design system: Tailwind theme tokens, radius/elevation/motion scales,
  dark-mode consistency, shadcn/ui primitives initialized and in use.
- Complete UI/UX redesign of every existing surface (marketing, auth,
  purchase, customer app, admin, emails, PDFs). No new product surfaces —
  this is polish, not expansion.
- Empty / loading / error / offline states for every interactive surface.
- Motion: Framer Motion for page transitions, list animations, and
  feedback microinteractions — used tastefully, never gratuitously.
- Accessibility to WCAG 2.2 AA on every public and authenticated surface.
- Performance: meet the budgets in §8 on 4G mobile.
- Mobile-first responsive audit: every surface looks right at 360 × 640.
- Expanded test coverage targeting the gaps the QA sweep surfaces.
- Visual regression baselines for the seven primary routes (new tooling:
  Playwright screenshot diffs).

**Out of scope** (explicitly)

- New product features — no new routes, no new data models, no new Inngest
  functions. If you discover a missing feature, write it up in the bug
  register and propose it for V1.2.
- Marketing content authored by a human (real testimonials, case studies,
  press logos). Design the slots; leave realistic placeholder copy that
  the operator will replace before launch. Do not fabricate attributions.
- Internationalization / translation.
- Migration away from the ADR-locked stack choices.
- Any change to the Tier 1 scope-disclosure language (master prompt §8
  requires verbatim).

## 5. Design system

### 5.1 Principles

1. **Restraint first.** Default to quiet. Color and motion emphasize
   meaning, never decoration. If a visual element doesn't change a
   customer's decision or confidence, delete it.
2. **Trust over delight.** This is a tax product. Errors in trust are
   terminal; errors in delight are recoverable. Do not make claims the
   product cannot back.
3. **Typography carries the experience.** We use two faces: Geist
   (interface) and Geist Mono (numeric). Big, confident headings;
   comfortable line-heights; no more than three type sizes per view.
4. **Motion is feedback, not decoration.** Use motion to confirm
   actions, bridge state changes, and reveal hierarchy. Disable on
   `prefers-reduced-motion`.
5. **One primary action per view.** Secondary actions are visibly
   subordinate. If two actions feel equal, the view is wrong.

### 5.2 Tokens (land in `app/globals.css`)

```css
@theme inline {
  /* Radii */
  --radius-xs: 4px;
  --radius-sm: 6px;
  --radius-md: 8px;
  --radius-lg: 12px;
  --radius-xl: 16px;
  --radius-2xl: 24px;
  --radius-full: 9999px;

  /* Elevation */
  --shadow-sm: 0 1px 2px rgba(0, 0, 0, 0.04);
  --shadow-md: 0 4px 12px -2px rgba(0, 0, 0, 0.06);
  --shadow-lg: 0 16px 48px -8px rgba(0, 0, 0, 0.12);

  /* Motion */
  --motion-fast: 120ms;
  --motion-base: 200ms;
  --motion-slow: 320ms;
  --ease-standard: cubic-bezier(0.2, 0, 0, 1);
  --ease-emphasized: cubic-bezier(0.3, 0, 0, 1);

  /* Semantic colors (overridden in dark mode) */
  --color-surface: var(--color-white);
  --color-surface-muted: #fafafa;
  --color-surface-raised: var(--color-white);
  --color-border: #e4e4e7;
  --color-border-strong: #d4d4d8;
  --color-foreground: #0a0a0a;
  --color-foreground-muted: #52525b;
  --color-foreground-subtle: #a1a1aa;
  --color-accent: #0a0a0a;
  --color-accent-foreground: #fafafa;
  --color-success: #047857;
  --color-warning: #b45309;
  --color-danger: #b91c1c;
}
```

Dark mode mirrors with the surface / foreground pair inverted and a
slight warm shift on borders.

### 5.3 Typography scale

- `display`: 48px / 56px tight, `Helvetica-Bold` / Geist Semibold
- `h1`: 36px / 44px, Semibold
- `h2`: 24px / 32px, Semibold
- `h3`: 18px / 24px, Semibold
- `body-lg`: 17px / 28px, Regular
- `body`: 15px / 24px, Regular
- `body-sm`: 13px / 20px, Regular
- `caption`: 11px / 16px, Regular, uppercase tracking `0.08em`

Numeric UIs use `Geist Mono` at the matching size. Tabular figures on
all money.

### 5.4 Spacing

Use Tailwind's default 4px grid. The only bespoke rule: sections have
`py-16` on mobile, `py-24` on desktop; section copy max-width
`max-w-2xl`; content max-width `max-w-5xl`.

### 5.5 shadcn components to add (list is authoritative)

`button`, `input`, `label`, `textarea`, `select`, `checkbox`, `radio-group`,
`switch`, `badge`, `card`, `separator`, `alert`, `alert-dialog`,
`dialog`, `dropdown-menu`, `tabs`, `tooltip`, `toast` (sonner),
`skeleton`, `progress`, `avatar`, `sheet`, `table`, `form`, `popover`,
`command` (cmdk), `navigation-menu`, `breadcrumb`, `scroll-area`.

Every primitive must be imported from `@/components/ui/*` and consume
tokens only.

### 5.6 Motion patterns

- Page enter: `opacity: 0 → 1` over `motion-base`, `ease-standard`.
- List stagger: 60 ms per child, max 6 children, falls back to static
  above 6.
- Modal: scale `0.96 → 1` + opacity over `motion-base`.
- Toast: slide-up 8 px + opacity.
- All animations respect `prefers-reduced-motion: reduce`.

## 6. Per-surface UX specs

Each surface has the same outline: **goal, layout, primary action,
states (loading/empty/error), motion, mobile, a11y, acceptance.**

### 6.1 Landing (`/`)

- Goal: convince the right visitor to run an estimate or scroll to
  pricing in under 15 seconds.
- Layout: Hero (value prop + two CTAs) → HowItWorks (three steps) →
  Estimator → SocialProof slots → PricingSection teaser → FAQ teaser
  → Footer. Sticky header; scroll-spy highlights active section.
- Primary action: "Estimate your savings" (anchors to estimator).
- Empty states: no dynamic content on `/`; do not render skeletons
  for the estimator — it's synchronous.
- Motion: Hero text fades + slides up 8 px on mount; pricing cards
  hover-lift 4 px.
- Mobile: single column; CTA buttons full-width; sticky header
  condenses to logo + menu icon.
- A11y: semantic `<section>` with `aria-labelledby`; skip link to
  `#main`.
- Acceptance: LCP < 1.5 s on 4G; all sections reachable via keyboard.

### 6.2 Estimator (component within `/`)

- Goal: deliver a credible range in under 3 s, funnel to lead
  capture + pricing.
- Layout: two-column on desktop (form left, result right — both
  permanent after first submit); single-column on mobile.
- Inputs: address with Google Places (gracefully degrades), property
  type select, purchase price (masked currency input), optional
  advanced accordion for bracket override.
- Result: headline range, stacked depreciation-class bar, year-1
  vs. remaining basis chart, scope-disclosure card, CTAs to pricing
  and email capture.
- States: input (default), submitting (skeleton on result column),
  result, error (inline, with retry).
- Motion: result column fades in; numbers count up over 600 ms.
- Mobile: result appears below form; auto-scroll to result on submit.
- A11y: every input has an associated `<label>`; errors have
  `aria-describedby`; number count-up respects reduced motion.

### 6.3 Pricing (`/pricing`)

- Goal: turn a convinced visitor into a paying customer. Differentiate
  Tier 1 vs. Tier 2 without diminishing Tier 1.
- Layout: comparison table (features × tiers) + both CTAs above fold;
  FAQ accordion below.
- Primary action: "Start an AI Report" / "Start an Engineered Study".
- States: no dynamic data.
- Mobile: tiers stack; comparison collapses to a disclosure per row.

### 6.4 Get-started (`/get-started`)

- Goal: ready the customer for a successful Stripe checkout in under
  90 seconds.
- Layout: two-step wizard (Account → Property), progress bar,
  back/next navigation, inline validation.
- Primary action: "Continue to secure checkout" (terminal).
- States: step 1, step 2, submitting (button spinner + page blocking
  overlay), server error (alert at top), cancelled
  (banner + pre-filled form).
- Mobile: wizard inherits the landing pattern; primary button
  sticks to the bottom.

### 6.5 Sign-in (`/sign-in`)

- Goal: frictionless sign-in in under 30 s.
- Layout: single centered card; magic-link primary, Google
  secondary; "not configured" admonition when env missing.
- States: idle, sending, sent (illustration + "check your email"),
  error (form-level).
- Motion: state transitions fade 120 ms.

### 6.6 Dashboard (`/dashboard`)

- Goal: at-a-glance view of all studies and their status; one click
  to continue intake or download a finished report.
- Layout: page header with primary CTA "Start a new study"; study
  cards in a responsive grid (1/2/3 cols).
- Card: property address (title), tier + status pills, time
  micro-copy, inline primary action driven by status, subtle cost
  micro-copy.
- States: empty (illustration + CTA), loading (3 skeleton cards),
  partial (DB unreachable → reassurance message, not a crash).
- Motion: card stagger on mount; action button pending states.

### 6.7 Intake (`/studies/[id]/intake`)

- Goal: guide the customer through a painless property + document
  submission in under 10 minutes.
- Layout: header with study summary + progress bar; three-tab
  wizard (Property / Documents / Review).
- Documents: per-kind cards with drag-drop, thumbnail/preview,
  percent-complete during upload, remove action. Required vs.
  optional kinds distinguished.
- Review: summary view before auto-submit; shows "all set —
  processing starts in a moment" when complete.
- States: all combinations of (property complete? × docs
  complete? × processing triggered?); a clear visual state for each.
- Motion: tab transitions, upload progress, success confetti on
  all-complete (once).

### 6.8 Admin (`/admin/*`)

- `/admin` (pipeline): toggle between kanban (default) and table.
  Filter chips + search. Cards draggable between columns is **out
  of scope** — use action menus instead.
- `/admin/studies/[id]`: tabbed inspector per §4.F. Action panel
  sticky on desktop, bottom sheet on mobile.
- `/admin/engineer-queue`: card stack, drag-to-upload for the signed
  PDF, shows an SLA clock per card.

### 6.9 Email templates

- Consistent header (wordmark), body type (15 / 24), single
  primary CTA, minimal chrome. Max content width 560 px.
- Scope-disclosure caveat in `WelcomeEmail` and Tier 1
  `ReportDeliveredEmail`; omit caveat from Tier 2 delivery email.
- Dark-mode email: leave default light palette; email dark-mode
  rendering is inconsistent, so we ship light only.

### 6.10 PDF deliverable

- Font: load Geist via `Font.register` or fall back to built-in

- Cover layout: property photo thumbnail if present, bold tier label,
  study ID as footer chip.
- Scope disclosure: **verbatim** from `lib/pdf/disclosure.ts`, on

  landed; preserve).
- Appendix: ATG 13-element map.

## 7. Accessibility requirements (WCAG 2.2 AA)

- All interactive elements keyboard-reachable in a logical order.
- Visible focus indicator on every focusable element.
- Text contrast ≥ 4.5:1 for body, ≥ 3:1 for large text and UI
  components. Verify with the audit tooling in `docs/polish/`.
- Every form field has a programmatically associated label; error
  text is linked via `aria-describedby`.
- Every image conveys semantics via `alt`; decorative images use
  `alt=""`.
- ARIA roles only when the native element won't do; prefer semantic
  HTML.
- Modal focus trapping; `Escape` closes.
- `prefers-reduced-motion`: disable translate / scale / opacity
  animations; keep `display` / focus changes.
- Landmarks (`<header>`, `<main>`, `<nav>`, `<footer>`) present
  exactly once per page where applicable.
- Skip link "Skip to content" on every page.
- `axe-core` in the Playwright suite; no new violations.

## 8. Performance budget

- **LCP** ≤ 1.5 s on a 4G Fast connection (Lighthouse CI).
- **CLS** ≤ 0.05 on every primary route.
- **INP** ≤ 200 ms on every interaction.
- **JS budget**: landing page first-load JS ≤ 140 KB gzipped; any
  route ≤ 220 KB. Verify in the `next build` output.
- Server responses for a cold dynamic page ≤ 300 ms (TTFB) p50 on
  Vercel Edge / Node baseline.
- Fonts: Geist + Geist Mono preloaded; `font-display: swap`; subset
  to `latin`.
- Images: `next/image` everywhere; no raw `<img>` on customer
  surfaces.

## 9. QA checklists

Every item is a `- [ ]` that must be checked off before closing
Phase I.

### 9.1 Code quality

- Every `as any`, `as unknown as`, or disabled lint rule has a
  justifying comment or is removed.
- No server action exports a non-async value (Next requirement).
- Every Prisma transaction has an explicit name in the StudyEvent
  kind if it writes a timeline event.
- Every third-party failure mode (Stripe, Supabase, Resend, Anthropic,
  Inngest, Textract) has a log line **and** a user-facing fallback.
- No `console.log` in `main`; only `console.info/warn/error` with
  structured context.
- All `env` access goes through `lib/env.ts`; no direct
  `process.env.FOO` outside that file and the bootstrapping configs
  (`proxy.ts`, `next.config.ts`, etc.).

### 9.2 Security

- `SUPABASE_SERVICE_ROLE_KEY` is never imported into a file without
  `import "server-only"`.
- Webhook endpoints verify signatures before any DB write.
- Rate limits present on all public-write endpoints (estimator, lead,
  sign-in, checkout). Per-IP where possible, per-email where the
  action is email-scoped.
- `assertOwnership` called on every protected query that takes a
  user-supplied id. Add a lint rule or a guard test if feasible.
- Signed URLs: default ≤ 1 hour for admin reads, 7 days for
  customer deliverables.
- No PII in Sentry breadcrumbs (scrub at the integration layer).
- Content-Security-Policy header set; Sentry + PostHog + Supabase +
  Stripe + Google Maps + Inngest domains explicitly allowlisted.

### 9.3 Data integrity

- Every Stripe webhook path is idempotent (existing `stripeSessionId`
  uniqueness covers it; write a test).
- Every Inngest function step is idempotent (covered by AiAuditLog).
- Every orphan state is recoverable via the admin inspector: a
  FAILED study can be re-run; a stuck PROCESSING study can be
