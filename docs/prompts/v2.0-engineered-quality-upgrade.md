# v2.0 — Upgrade AI Report to Engineered-Study Quality

> Paste this whole file to Claude Code as one prompt. It is the master spec for
> rebuilding the AI cost-segregation report so it matches the quality of a
> $700 engineered deliverable (see `207SEdisonCostSeg.pdf` as the target).

---

## Context you need before touching code

We ship a Tier-1 AI-generated cost-segregation PDF from
`components/pdf/AiReportTemplate.tsx`, rendered by `lib/pdf/render.ts` and
delivered by `inngest/functions/deliver-ai-report.ts`. The current asset
classification runs in `lib/ai/steps/classify-assets.ts` with the prompt at
`lib/ai/prompts/classify-assets.ts`.

**The user drops three things into the app:**

1. Closing disclosure / HUD-1 / ALTA settlement (PDF).
2. A folder of property photos (interior + exterior).
3. A spreadsheet of capitalized improvements and furnishings with costs.

**What happens today (verified by reading the current report output):**

- The photos are never read by a model. The `uploads` are discarded after
  basis extraction.
- The improvements spreadsheet is consumed only as raw line items and
  bucketed into 5yr/15yr/27.5-39yr — it does not drive per-item classification.
- The classify-assets schema in `lib/ai/prompts/classify-assets.ts` caps at
  `maxItems: 40` and the model returns ~12 category-level buckets such as
  "Appliances — 3.00% of basis — $10,011." There is no specificity.
- `components/pdf/AiReportTemplate.tsx` line 1313 declares a "Property photos"
  reference card but no photo is ever embedded anywhere in the PDF.
- Property Description on the cover page reads "Square footage and year built
  were not provided." We never fetch a public listing/assessor record.
- Land allocation falls back to "Rule 3: 30% midpoint" instead of using the
  actual assessor land/total ratio.
- Appendix C is four abstract bullets. The actual closing disclosure and
  improvements spreadsheet are not rendered into the PDF.

**What the $700 benchmark does (read both PDFs side-by-side before starting):**

- Appendix B enumerates **~145 discrete line items**: "Chrome double towel
  bar," "Wooden shower bench," "Decorative hanging planters (qty 3)," "King
  bed frame & mattress," etc. Every line item has a thumbnail photo inline.
- Each item carries six documented fields: a Craftsman-/RSMeans-/PriceSearch
  comparable with retailer URL, a physical-condition multiplier with one
  paragraph of justification, a functional-obsolescence multiplier with one
  paragraph of justification, a time multiplier (Building Cost Historical
  Index), a location multiplier (Area Modification Factor), and a fully
  adjusted cost.
- Every improvement from the uploaded spreadsheet appears twice: once as its
  own Appendix B line item (with `Source: receipt` and `Time (1.0000):
Receipt-backed item uses recorded expenditure cost`) and once in the
  Appendix D classification schedule.
- Property Description pulls actual listing facts: 2,197 sq ft, built 1920,
  wood exterior, metal roof, detached garage, 7,884 sq ft lot.
- Land value uses the assessor ratio
  ($155,080 / $536,000 = 28.93%) applied to purchase price, with the source
  URL printed in the report.
- Appendix C embeds the closing disclosure and improvements spreadsheet as
  scanned pages at the back of the PDF.

**Accelerated depreciation delta that falls out of all this:** 29.0% vs
18.9% of depreciable basis. The pricing discipline alone buys real money
for the customer. This is the moat.

---

## Goal

Rebuild the AI report generation pipeline so that, given the same three
inputs, we produce a PDF that is visually and substantively indistinguishable
from the $700 engineered study for a non-expert reader — and that a reviewing
CPA/PE could sign off faster than they could from scratch.

Not a goal: claim PE-signed audit-defensibility. Keep the "Tier-1 planning
estimate, not an IRS-defensible study under Pub 5653" disclosure. The
improvement is quality of work product, not regulatory tier.

---

## Phased plan — do these in order, ship each one behind a feature flag

### Phase 1 — Photos become first-class inputs

1. Extend the upload schema and Prisma models so a study has an ordered
   list of photos with: file URL, image dimensions, original filename, an
   optional user-supplied room tag, and two model-generated fields
   (`caption`, `detectedObjects`). Run a migration.
2. Add `lib/ai/steps/describe-photos.ts`. For each photo, call the vision
   model with the photo and a prompt that asks it to:
   - Identify the room type (kitchen, primary bath, exterior front, etc.).
   - List every discrete depreciable object visible, one per line, with a
     short specific name ("chrome double towel bar above toilet", not
     "bathroom fixtures") and a category tag from our asset taxonomy.
   - For each object, assess visible condition on the Excellent/Good/Fair/
     Poor/Salvage scale with a one-sentence justification rooted in what is
     actually visible ("no scratches, finish intact, appears like new").
   - Return structured JSON via tool use; enforce with Zod.
3. Persist the per-photo detected-object list. This is the spine of the new
   asset schedule.

### Phase 2 — Asset schedule built from detected objects + receipts, not category percentages

1. Rewrite `lib/ai/prompts/classify-assets.ts`. The new prompt consumes:
   - All detected objects from Phase 1 (deduplicated across photos — the
     same dining table seen from two angles is one line item).
   - The improvement-spreadsheet line items (each becomes a
     `basis: "itemized_receipt"` line).
   - The closing-disclosure capitalized-cost items (title insurance,
     recording, escrow — each becomes a `basis: "itemized_receipt"` line,
     classified 39yr unless a specific rule says otherwise).
   - The property-type asset library as a sanity-check, not a template.
2. Raise `maxItems` to `300`. Realistic target is 80–180 line items per
   property.
3. Extend `assetLineItemSchema` with the six adjustment fields the
   engineered study carries:
   - `source: "craftsman" | "rsmeans" | "pricesearch" | "receipt"`
   - `comparable: { description, unitCost, unit, sourceUrl? }`
   - `physicalMultiplier` + `physicalJustification` (paragraph)
   - `functionalMultiplier` + `functionalJustification` (paragraph)
   - `timeMultiplier` + `timeBasis` (e.g. "Building Cost Index
     2025→acquisition year")
   - `locationMultiplier` + `locationBasis` (e.g. "Austin, TX Area
     Modification Factor 1.0900")
   - `photoRef` pointing back to the source photo (null when the item
     comes from a receipt with no photo match).
4. The balancing constraint changes: **no longer force line items to sum
   exactly to building value.** Instead, compute the sum of adjusted costs,
   let the reconciling "Building structure (residual)" 27.5-/39-year line
   absorb the remainder — exactly how the professional report handles it
   (see its "Used as the reconciling plug to hit exact building value"
   rationale).
5. Keep the validator, but teach it to check per-category caps from the
   asset library and to flag any adjusted-cost outliers (e.g. a towel bar
   priced at $500).

### Phase 3 — Pricing sources

Decide honestly what we have. Three options, pick one per source:

- **If we license Craftsman / RSMeans / PriceSearch data:** wire up the
  lookup and return real unit costs and page references.
- **If we don't:** have the model produce a plausible comparable with a
  real retailer URL (Target/Wayfair/Lowe's/Home Depot) and label the source
  `"pricesearch"`. Do NOT fabricate Craftsman page numbers. The prompt must
  explicitly forbid inventing page references.
- **For receipts:** unit cost = receipt amount, multipliers all 1.0000,
  label `"receipt"`. This matches the benchmark's behavior.

Document the choice in `docs/adr/` and update the report's methodology
appendix to describe exactly which of the three sources is actually
consulted. Do not misrepresent our data sources — that's the one thing a
reviewer will catch.

### Phase 4 — Property facts from public data

1. Add `lib/ai/steps/enrich-property.ts`. Given the address, fetch from the
   county assessor record and a public listing source (Redfin/Zillow/Realtor)
   and extract: square footage, year built, bedrooms/baths, construction
   type, roof type, lot size, assessor land value, assessor total value.
2. Pipe these into Property Description prose and into the land allocation.
3. Land allocation priority order: (1) closing-disclosure allocation if
   present, (2) assessor land/total ratio applied to purchase price,
   (3) market-rule fallback. Always print the source URL.

### Phase 5 — Report template

Rewrite `components/pdf/AiReportTemplate.tsx` so that Appendix B becomes a
per-asset gallery. Each asset gets a page (or half-page) with:

- Photo thumbnail (~2.5" wide), left-aligned.
- Name, classification, category, quantity, description.
- Cost Estimate block with source, unit cost, unit, comparable description
  and URL.
- Per-Item Adjustments (physical + functional, each with the paragraph).
- Time Adjustment (factor + date basis).
- Global Adjustments (location factor + location string).
- Cost Summary (base cost × multipliers = fully adjusted cost).

Read the benchmark PDF pages 12–204 for the exact layout. Match spacing
and typography closely — the visual density is part of the perceived value.

Add new Appendix C that actually renders the source documents:

- Embed the closing disclosure as image pages (convert PDF → images →
  `<Image>` in react-pdf).
- Embed the improvements spreadsheet as a rendered table (or as a
  screenshot if it came in as an image).
- Embed any property-photo pages not shown in Appendix B as a photo
  index.

Target page count: **150–220 pages** for a typical property, versus
today's 31.

### Phase 6 — Polish

- Update the cover page so Property Description pulls real facts.
- Update Executive Summary so the accelerated-property % reflects the
  new schedule (should land in the 22–32% range for most STRs).
- Keep the "Planning estimate, not an IRS-defensible study" footer on
  every page — do not remove the disclosure.
- Update `tests/unit/pdf-render.test.ts`, `asset-library.test.ts`, and
  `classify-assets-prompt.test.ts` to cover the new shape. Snapshot a
  fixture study end-to-end.

### Phase 7 — QA gate (deterministic + vision-model review)

Ship this as a pipeline step inside the existing Inngest flow, not as a
separate managed agent. Two layers:

**7a. Deterministic layout discipline in the template.**

- Wrap every atomic block (asset line-item card, executive-summary panel,
  MACRS table row group) in `<View wrap={false}>` so react-pdf can never
  split the block across a page break.
- Use `<Text minPresenceAhead={N}>` on every h2/h3 so a section heading
  cannot orphan at the bottom of a page.
- Explicit `break` props before major appendix starts.
- Add `tests/unit/pdf-layout.test.ts` that walks react-pdf's rendered
  tree and asserts invariants:
  - No `<Image>` element has bounding-box overlap with any `<Text>`
    element on the same page.
  - Every page contains the disclosure footer text.
  - Every h2 appears on the same page as at least one following paragraph.
  - No page ends with fewer than 3 lines of a paragraph that continues.

**7b. Vision-model review step, as the last Inngest step.**

1. After `render.ts` produces the PDF, rasterize every page to PNG
   (`pdf-to-png-converter` or pdftoppm) at ~150 DPI. Persist them to
   temporary storage keyed by study ID.
2. New step `lib/ai/steps/review-report.ts` that calls Claude with vision
   on the page images (batched, ~10 pages per call to stay under the
   context ceiling) with a structured rubric:
   - **Layout blockers**: image overlapping text, text cut off at page
     edge, table column misaligned, heading orphaned at page bottom,
     asset card split across pages.
   - **Content blockers**: placeholder text visible ("TBD", "Not
     specified", "[insert here]", lorem ipsum), a line item missing its
     photo or missing one of the six adjustment fields, disclosure
     footer absent on any page, Appendix B truncated, totals in
     Executive Summary not matching Appendix D.
   - **Warnings**: prose reads awkwardly, two consecutive pages are
     visually near-identical, photo quality looks too low to be useful.
   - **Nits**: minor typography issues, spacing inconsistencies.
3. Output via tool use: array of `{ page, severity, category, message,
suggestedFix }` with a Zod schema.
4. Branching logic:
   - Any blocker → feed the issue list back into the upstream step it
     came from (layout → re-render with fix hints; content →
     classify-assets with priorAttemptError), with a retry cap of 2.
     If still blocked after 2, write a `ReportReviewFailure` record and
     surface it to internal ops — do not ship to the user.
   - Only warnings/nits → log to telemetry, ship the PDF.
5. Keep the review prompt and rubric versioned next to the template. When
   you change `AiReportTemplate.tsx` you'll often need to update what the
   reviewer looks for — atomic PRs.

**Why a pipeline step, not a Claude Console managed agent.** The
reviewer needs the rendered PDF bytes and page images, its output feeds
straight back into regeneration in the same flow, and the rubric
references this template's exact structure. Crossing an API boundary to
a managed agent means uploading artifacts to a signed URL, round-tripping
results, and living with two independent deploys of the rubric vs. the
template. That's worth it for a user-facing shared agent; it's overhead
for an internal quality gate. Keep everything in-repo, versioned
atomically with the template.

## Specific prompt-engineering guidance

When you rewrite `lib/ai/prompts/classify-assets.ts`:

- Open with the role: "You are a senior cost-segregation engineer
  assembling a per-item asset schedule by examining property photographs."
- Make the photo→objects→line-items pipeline explicit in the system
  prompt. Tell the model: for each detected object, produce exactly one
  line item; do not aggregate; specificity is the product.
- Give two or three in-context examples of the target output quality,
  drawn verbatim from the benchmark. The "Decorative Hanging Planters,"
  "Chrome Double Towel Bar," and a receipt-backed example like "Queen
  Bed Frame & Mattress (Guest)" are ideal.
- Forbid: inventing unit costs without a source, fabricating Craftsman
  page numbers, generic descriptions ("bathroom fixtures"), photos not
  actually in the input.
- Require: cents-exact arithmetic, URL or source label on every comparable,
  multiplier values in the range the condition scale allows, a one-sentence
  rationale per line item that would satisfy a reviewing engineer.

## Acceptance criteria (run these before marking done)

On the Edison Street fixture (same inputs the benchmark used):

1. PDF has ≥80 Appendix B line items, each with an embedded photo or a
   `Source: receipt` label.
2. Every improvement-spreadsheet row appears as its own Appendix B line
   item AND in Appendix D.
3. Property Description contains square footage, year built, and
   construction type — no "Not specified."
4. Land allocation uses the assessor ratio, not the 30% rule.
5. Appendix C embeds the closing disclosure and improvements spreadsheet
   as rendered pages (not prose placeholders).
6. Accelerated property share is within ±3 percentage points of the
   benchmark (benchmark: 29.0%).
7. Side-by-side visual diff against `207SEdisonCostSeg.pdf`: a stranger
   flipping through both should not be able to pick which one came from
   an AI in under 30 seconds.
8. Disclosure footer still says "Planning estimate, not an IRS-defensible
   study under Pub 5653."
9. The vision-model review step runs on the fixture and returns zero
   blockers; the layout unit test passes (no image/text overlap, no
   orphaned headings, disclosure on every page).

## Non-negotiables

- Do not claim PE sign-off we don't have.
- Do not fabricate citations (Craftsman page numbers, RSMeans references,
  assessor URLs). If the source isn't real, label the line `pricesearch`
  with the real retailer URL.
- Keep the per-page Tier-1 disclosure.
- Keep the existing basis/depreciable-basis/MACRS math exactly as it is —
  the upgrade is the asset schedule and surrounding prose, not the
  depreciation engine.
